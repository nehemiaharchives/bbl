# Nix/NixOS Package Support

## Overview

The bbl CLI can be built and installed using the Nix package manager. Unlike `.deb`/`.rpm`/Homebrew packages, the Nix package is distributed as a **flake tarball** — not a standalone archive file. Users download the tarball, extract it, and run `nix build` / `nix profile install` from the flake.

The Nix package is tested using a **direct Docker approach** (`nixos/nix:latest` image), not Chef/Test Kitchen. See [Testing](#testing) below.

## How the Nix Package Works

### Wrapper Script

The bbl binary expects translation packs at `$HOME/.bbl/packs/<code>.zip` and search helpers at `$HOME/.bbl/bin/bbl-search-*`. The Nix derivation bundles these files in the store, and the generated **wrapper script** (`$out/bin/bbl`) handles the first-run copy:

1. User runs `bbl` → wrapper script executes
2. Wrapper checks if `$HOME/.bbl/bin/bbl-search-common` and `$HOME/.bbl/packs/webus.zip` exist
3. If missing, copies them from the Nix store (`libexec/bbl/`, `share/bbl/packs/`)
4. Then `exec`s the real binary (`libexec/bbl/bbl`)

```bash
# Generated wrapper at $out/bin/bbl:
set -euo pipefail
assets="$(cd "$(dirname "$0")/.." && pwd)"
mkdir -p "$HOME/.bbl/bin" "$HOME/.bbl/packs"
if [ ! -f "$HOME/.bbl/bin/bbl-search-common" ]; then
  install -m 0755 "$assets/libexec/bbl/bbl-search-common" "$HOME/.bbl/bin/bbl-search-common"
fi
if [ ! -f "$HOME/.bbl/packs/webus.zip" ]; then
  install -m 0644 "$assets/share/bbl/packs/webus.zip" "$HOME/.bbl/packs/webus.zip"
fi
exec "$assets/libexec/bbl/bbl" "$@"
```

After the first run, `$HOME/.bbl/` has everything bbl expects — identical to what the `.deb`/`.rpm`/Homebrew wrappers do.

### Runtime Closure

The wrapper script references `${coreutils}/bin/install` (for file copying). Nix automatically detects this store path reference in the output file and includes `coreutils` in the runtime closure. The `nix build` and `nix profile install` commands ensure all closure paths are available.

### `bbl install kjv` (installing additional translations)

When the user runs `bbl install kjv`, bbl downloads `kjv.zip` from the GitHub release URL (`https://github.com/nehemiaharchives/bbl/releases/download/<version>/kjv.zip`) and saves it to `$HOME/.bbl/packs/kjv.zip`. This is pure Kotlin code — identical behavior across all platforms (deb, rpm, Homebrew, Nix, bare binary). If the release asset exists, it downloads successfully; if not (e.g., dev/bleeding-edge version not yet published), it returns HTTP 404.

## Fixture Structure

Generated by `./gradlew stageBblInstallLinuxNixFixture` into `build/bblInstallFixtures/linux/nix/`:

```
bbl-nix/
├── flake.nix          # Nix flake definition
├── bbl.nix            # Package derivation
├── assets/
│   ├── bbl            # Release bbl binary (stripped, ~15 MB)
│   ├── bbl-search-common  # Search helper binary (~22 MB)
│   └── webus.zip      # Bundled Webus translation
└── README.md
```

Published as `bbl-<tag>-linux-x64-nix-flake.tar.gz` release asset.

## Testing

The Nix package is tested using `nixos/nix:latest` Docker image, not Chef/Test Kitchen.

### Why Docker instead of Test Kitchen?

- Test Kitchen is Chef/Ruby-based, designed for system packages (.deb, .rpm)
- Nix requires `nix-daemon` — Test Kitchen instances would need Nix installed inside the VM
- `nixos/nix:latest` provides a ready-to-use Nix daemon and store out of the box

### Local Testing

```bash
# 1. Generate the fixture (release build)
./gradlew stageBblInstallLinuxNixFixture -PExecutableType=release -Pbblpacks.embed=false

# 2. Test with Docker (no local Nix needed)
docker run --rm -it \
  -v $PWD/build/bblInstallFixtures/linux/nix:/fixture:ro \
  nixos/nix:latest bash
```

Inside the container:

```bash
# Extract and build
tar xzf /fixture/bbl-nix.tar.gz -C /tmp
cd /tmp/bbl-nix/bbl-nix
nix --extra-experimental-features "nix-command flakes" build .#bbl --out-link /tmp/result

# Run tests
/tmp/result/bin/bbl --version
/tmp/result/bin/bbl john 3:16
/tmp/result/bin/bbl search God limit 1

# Test profile install
nix profile install --profile /tmp/profile .#bbl
/tmp/profile/bin/bbl john 3:16
```

### Debug vs Release

```bash
# Debug build (larger binaries, debug symbols)
./gradlew stageBblInstallLinuxNixFixture -PExecutableType=debug -Pbblpacks.embed=false

# Release build (stripped, optimized)
./gradlew stageBblInstallLinuxNixFixture -PExecutableType=release -Pbblpacks.embed=false
```

The fixture structure is identical for both — only the embedded binaries differ.

## CI Pipeline

See `.github/workflows/ci.yml`:

- Matrix entry: `installer-nix` (platform: nixos)
- E2E entry: `nix_install_nixos` (fixture_pattern: bbl-install-linux-nix-files)
- Docker E2E step runs `nix build`, `bbl --version`, `bbl john 3:16`, `bbl search`, `nix profile install`

## Publishing

See `.github/workflows/publish.yml`:

- `stageBblInstallLinuxNixFixture` is called in the linux-x64 matrix entry
- The flake tarball (`bbl-<tag>-linux-x64-nix-flake.tar.gz`) is copied during "Assemble Unix release assets"

## Known Issues

### `libcrypt.so.1` → `libcrypt.so.2`

Kotlin Native binaries link against `libcrypt.so.1`. Nixpkgs `libxcrypt` only provides `libcrypt.so.2`. The `bbl.nix` derivation uses `patchelf --replace-needed` in `preFixup` to remap the dependency:

```nix
preFixup = ''
  patchelf --replace-needed libcrypt.so.1 libcrypt.so.2 $out/libexec/bbl/bbl
  patchelf --replace-needed libcrypt.so.1 libcrypt.so.2 $out/libexec/bbl/bbl-search-common
'';
```
